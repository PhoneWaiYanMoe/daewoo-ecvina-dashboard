Overall, the three Daewoo E&C Vina web servers hosted on TechValley / CMC Cloud operated within safe limits throughout June 2026, with no downtime-causing incidents recorded in the monitoring data. CPU, memory and load metrics stayed comfortably below critical thresholds on all hosts for the majority of the month.

web.boatcon (construction site portal) showed the lightest and most stable load of the three: CPU averaged 5.33% (peak 5.63%), memory 33.67% (peak 41.76%), and disk usage a steady ~38%. This host has significant spare capacity.

web.boatnet is the outlier and the main point requiring attention. Filesystem usage averaged 77.46% and peaked at 86.98% — this is approaching the 80–90% range where write failures, log-rotation issues, or database/application errors typically begin to appear. In addition, this host showed a sharp CPU spike (57.88% vs. a 1.10% average) and a large outbound traffic burst (185.11 Mbps vs. a 954 kbps average) occurring well outside its normal baseline. The combination of a CPU spike, network burst, and disk-IO spike (10.75% vs. 0.08% avg) in the same dataset is consistent with a scheduled batch job, backup/replication run, or a large file transfer — not a chronic overload — but it should be confirmed and scheduled outside business-critical hours if not already.

web.boatsale (public sales portal) ran with healthy headroom: CPU under 3.5%, memory around 57%, and only 12.6% filesystem usage — the most disk headroom of the three servers, but the highest average memory utilization, worth tracking as traffic grows.

No server exceeded CPU, memory or load thresholds on a sustained basis, and all "Max" spikes observed were short-duration (1-hour average bucket) events rather than persistent saturation.

The web security service (WEBCastle, WSFinder, WMDS) performed well overall in June, blocking 35,367 attacks at a 99.99% block rate — only 2 events were detected but not auto-blocked, and none resulted in a confirmed compromise. This was the highest attack volume of the last 6 months, up 4,942 cases from May, with a sharp single-day spike of 12,370 blocked attacks on June 1 and a secondary spike near month-end.

Nearly all traffic (79.4%) was flagged as "Standard Protocol" violations and 18.6% as LFI (Local File Inclusion) attempts — both fully blocked by the firewall. Risk classification split 62.3% Medium and 37.7% High, with no Critical incidents. Under OWASP Top 10, essentially all classified attacks fall under A04: Insecure Design, which is typical of broad, automated scanning rather than a targeted campaign exploiting a specific known bug.

The attack traffic is highly concentrated: the top 5 attacker IPs all sit inside the same 185.177.72.0/24 block, each producing ~3,000 hits — almost certainly one coordinated scanning source, not five independent attackers. Nearly all of it (35,120 of 35,367 attacks) was aimed at the public web.boatsale portal. Cross-referencing with the performance data is reassuring: despite absorbing essentially all of this month's attack traffic, web.boatsale's CPU, memory and disk stayed low and stable all month — the firewall is filtering this traffic before it reaches the application. Correspondingly, the performance spike seen on web.boatnet is not attack-related, since that host was barely targeted at all — it reinforces the earlier conclusion that the boatnet spike was a scheduled internal job, not a security event.

Webshell scanning (WSFinder) and malware-distribution scanning (WMDS) both came back clean this month: 0 webshells across 775 scanned files, and 0 malware detections across 31 daily-scanned URLs. 25 URLs were flagged for general inspection, 80.65% of them due to timeouts — worth a quick operational check, but not indicative of a security breach.
